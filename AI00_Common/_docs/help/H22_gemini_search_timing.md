## 基本原则

Gemini 有两个信息来源：

| 来源 | 内容 | 例子 |
|------|------|------|
| 自己的知识（训练数据）| 训练截止前的信息 | 历史、常识、编程知识 |
| Google 搜索（实时）| 训练截止后 / 实时变化的信息 | 今天价格、最新新闻、票务库存 |

---

## 同一对话内：第二次不用搜索

```
第1次提问：「Mac mini M4 在 Amazon 多少钱？」
→ Gemini 搜索 Google → 找到「128,000 日元」
→ 这个结果存在对话上下文里

第2次提问：「比 Apple Store 贵吗？」
→ Gemini 已经知道 Amazon 价格（上下文里有）
→ 不需要再搜索 Amazon
→ 只需要查 Apple Store 价格
```

**同一对话内已经搜索过的信息，Gemini 会记住，不重复搜索。**

---

## 新的对话 / 新的 API 调用：重新搜索

```
PJ90 监控演唱会票（每10分钟检查一次）：

第1次检查（10:00）→ 新的 API 调用 → Gemini 搜索 → 「没票」
第2次检查（10:10）→ 新的 API 调用 → Gemini 搜索 → 「没票」
第3次检查（10:20）→ 新的 API 调用 → Gemini 搜索 → 「有票！」→ 推送通知
```

**每次 Firebase 定时触发 = 新的 API 调用 = Gemini 重新搜索**
实时监控必须每次都搜索，确保信息是最新的。

---

## 总结

| 情况 | 用 Google 搜索？ |
|------|----------------|
| 同一对话，已搜索过的信息 | ❌ 不用，用上下文 |
| 新对话 / 新 API 调用 | ✅ 重新搜索 |
| Gemini 自己知道的知识 | ❌ 不用搜索 |
| 实时价格 / 库存 / 新闻 | ✅ 每次搜索 |

---

## 对 PJ90 的影响

- 用户和 Gemini 聊天时：同一对话内搜索结果共享，效率高
- PJ90 定时监控任务：每次都是新调用，每次都搜索，保证实时性
- 费用：搜索次数 = API 调用次数，但 Gemini 免费额度足够个人用
