# H37 — 规则优先级指南（L1-L4）

> 给人看。AI 遵守规则的顺序和原因。

---

## 为什么要有优先级？

规则越来越多，有时候会互相冲突。有了优先级，冲突时知道听哪个。

**总原则**：L1 最高，L4 最低，高级别规则不可违反。

---

## 四个级别

### L1 — Claude Code 效率（最高，不可违反）

> **目标**：每次会话 AI 能快速进入状态，不浪费 token。

| 规则 | 为什么 |
|------|--------|
| 自动加载 ≤ 5k tokens | 超过这个 AI 开始忽略规则 |
| 项目规则放项目 CLAUDE.md | 不相关项目的规则不应该占用空间 |
| 大功能用 skills 按需加载 | 只在需要时加载，不常驻 |
| 跳过无关文件夹 | 避免搜索到历史/存档内容 |

**⚠️ 业界警告**：CLAUDE.md 超过 200 行，规则开始被忽略；超过 500 行，AI 基本不遵守。T00 目标：CLAUDE.md < 150 行。

### L2 — Reply 优先（不可省略）

> **目标**：每次回答都完整写入 reply，CLI 清空后下次还能继续。

| 规则 | 为什么 |
|------|--------|
| 实质性回答必须写进 reply | CLI 随时可能被清空 |
| CLI 只输出简短确认 | 不在 CLI 重复 reply 内容 |
| Reply 含原文 + 时间 + 行号 | 下次会话也能看懂 |
| Reply 内容必须完整可执行 | 不能写"下一步再做"，要直接做 |
| 重要内容会话结束前保存 | Reply 不是永久存储 |

### L3 — T00 框架（工作流规范）

> **目标**：保证工作流的一致性和可追溯性。

- 项目内容存项目文件夹（不散落根目录）
- 说「再见」自动存档（SESSION_LOG + RESUME）
- 说「建项目」自动分配编号并注册
- 说「以后想做」记录到 FUTURE_FEATURES，不立即实现
- `[手动修改]` 标注区域绝对不修改
- 清空操作必须征得用户同意

### L4 — 开发规范（有条件执行）

> **目标**：保证代码质量和安全，但不影响效率。

- trust: high → 修改前说明理由
- 重大变更前先 commit
- 自动安装工具：仅用户明确遇到问题时触发（不主动搜）
- 小任务直接做，大任务（5+文件）派子 Agent

---

## 冲突举例

**例1**：AI 想自动安装一个 npm 包（L4）→ 但这会多读文件影响 token（L1）
→ **L1 优先**，不主动安装，等用户说"装 xx 包"

**例2**：用户说"下次再写 help 文件"（L3延后）→ 但 reply 里写了"下一步做 H37"（L2要完整）
→ **L2 优先**，当场做完，不留"下一步"
