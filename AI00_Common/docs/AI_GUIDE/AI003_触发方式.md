# AI003 — 触发方式（告诉 Claude 要做什么的四种方法）

> 根据你的状态选择最合适的方式，减少沟通成本。

---

## 四种方式一览

| 方式 | 你的状态 | 你做什么 | Claude 做什么 |
|------|---------|---------|-------------|
| **A** IMPROVE_REQUEST | 知道要什么（多项）| 填模板文件 | 按清单逐项实现 |
| **B** 直接口头 | 知道要什么（单项）| 直接说 | 立即实现 |
| **C** 设计书对比 | 知道怎么改 | 改设计书文件 | 按新设计实现 |
| **D** INBOX | 只有模糊想法 | 写想法 | 先设计方案→你确认→再实现 |

---

## 方式 A — IMPROVE_REQUEST（多需求推荐）

**文件路径**：`docs/REQUESTS/IMPROVE_REQUEST.md`

**步骤**：
```
1. 打开 IMPROVE_REQUEST.md
2. 按模板填写要做的内容
3. 告诉 Claude："IMPROVE_REQUEST 填好了，请实现"
```

**适合**：有多个改进点、需要 Claude 统一规划实现。

**示例写法**：
```markdown
## 请求日期
2026-06-01

## [页面名]
1. 添加XXX功能
2. 修改YYY的显示方式

## 全局
1. 导航栏加深色模式切换
```

---

## 方式 B — 直接口头（简单修改推荐）

**步骤**：
```
直接在对话里说要做什么
```

**适合**：单一、简单的改进或 Bug 修复。

**示例**：
```
在「复习」页面，切换标签后卡片列表没有刷新。
期望：切换后只显示该标签的卡片。
```

---

## 方式 C — 设计书对比法（大功能推荐）

**步骤**：
```
1. 打开对应的设计书文件
2. 直接修改成你想要的样子
3. 告诉 Claude："我改了[文件名]，请按新设计实现"
```

**适合**：功能结构变化较大，需要系统性修改。

**对应关系**：

| 要改的内容 | 改哪个设计书文件 |
|---------|--------------|
| 某个页面的交互/布局 | 界面说明文件 |
| 某个功能的行为逻辑 | 功能说明文件 |
| 数据库表结构 | 数据模型文件 |
| API 端点 | API接口文件 |

---

## 方式 D — INBOX（想法阶段推荐）

**文件路径**：`docs/REQUESTS/INBOX.md`

**步骤**：
```
1. 打开 INBOX.md
2. 在「想法区」随意写下想法（越随意越好，不需要完整）
3. 告诉 Claude："INBOX 有新内容"
```

**Claude 的流程**：
1. 读取想法内容
2. 提出 2-3 个实现方案（含优缺点）
3. 等你选择或补充反馈
4. 确认后才开始设计书更新和代码实现

**适合**：你有方向感但还不清楚具体要怎么做时。

**示例写法**（不需要完整，随意写）：
```
想法：感觉导入功能有点麻烦，能不能更简单一点
想法：加个统计图表，让进度更直观
想法：手机上用起来不太顺手
```

---

## 如何选择方式

```
有多个明确需求？→ 方式 A（IMPROVE_REQUEST）
只有一个明确需求？→ 方式 B（直接说）
知道要改哪里但想让设计先对齐？→ 方式 C（改设计书）
只有模糊的感觉/方向？→ 方式 D（INBOX）
```
