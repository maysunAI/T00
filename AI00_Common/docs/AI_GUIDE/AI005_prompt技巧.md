# AI005 — Prompt 写法技巧（让 Claude 更准确理解你的意图）

> 同样的需求，说法不同，结果差很多。本文列出经过验证的高效说法。

---

## 1. 核心原则：说结果，不说步骤

| ❌ 低效说法 | ✅ 高效说法 |
|-----------|-----------|
| 你先读这个文件，然后找到函数，再修改它 | 把「登录页面」的按钮颜色改成蓝色 |
| 我觉得可能是数据库的问题，你去看看 | 登录时报错：`SQLite: no such table`，帮我找原因 |
| 你懂吗，就是那个功能，改一下 | 把「用户列表」页面的分页改成每页 20 条 |

**关键**：说你想要什么，不说 Claude 应该怎么做。

---

## 2. 五要素模板（复杂需求必用）

```
页面/模块：[具体位置]
现状：[现在是什么情况]
期望：[想要变成什么样]
原因：[为什么要改（可选）]
边界：[什么情况不要改（可选）]
```

**示例**：
```
页面/模块：用户设置页面
现状：保存按钮点击后没有任何提示
期望：保存成功后显示绿色提示"保存成功"，3 秒后消失
边界：不要修改按钮本身的样式
```

---

## 3. Bug 报告（附错误信息 = 速度 ×10）

```
现象：[描述用户看到的问题]
错误信息：
[从浏览器 F12 控制台 / 终端 直接粘贴]

复现步骤：
1.
2.
3.
```

> 错误信息比你的描述更重要。不确定哪里出错时，把 F12 全部粘贴。

---

## 4. 控制输出范围（防止 Claude 改太多）

| 想要的限制 | 怎么说 |
|---------|------|
| 只改这一个地方 | 「只修改 [文件名] 的 [函数名]，不要动其他地方」 |
| 不要加新功能 | 「只修复这个 Bug，不要顺便重构」 |
| 先分析再动手 | 「先分析原因，不要修改代码，等我确认后再改」 |
| 分步执行 | 「先做第一步，完成后告诉我，我确认后再做第二步」 |

---

## 5. 获取方案选项（不确定怎么做时）

```
我想做[功能]，帮我提出 2-3 种实现方案，各说优缺点，先不要动代码。
```

Claude 会列出方案，你选一个，再说「用方案 2 实现」。

---

## 6. 让 Claude 解释代码（理解现有代码）

| 目的 | 说法 |
|-----|-----|
| 理解整体流程 | 「解释一下 [功能] 的实现流程，用大白话，不需要逐行解释」 |
| 找到某个功能 | 「[功能] 是在哪个文件/函数实现的？」 |
| 理解一段代码 | 「解释这段代码做了什么：[粘贴代码]」 |
| 确认是否正确 | 「[函数名] 的逻辑是否正确？预期：[描述]」 |

---

## 7. 规则设定（写进 CLAUDE.md 永久生效）

```
以后每次[触发条件]，你都要[具体行为]。把这个规则写进 CLAUDE.md。
```

**示例**：
```
以后每次我说"检查一下"，你都要列出可能有问题的地方但不修改。把这个写进规则。
```

> 规则写进 CLAUDE.md 后，每次新对话自动生效，不需要重复说。

---

## 8. 常见失败原因 & 修复

| 问题 | 原因 | 修复说法 |
|-----|-----|--------|
| Claude 改了太多地方 | 需求范围不清 | 加「只修改 XXX」 |
| Claude 反复问确认 | 信息不够具体 | 补充页面名、功能名、期望结果 |
| Claude 做了一半停了 | 任务太大 | 说「先做第一步，完成后告诉我」 |
| Claude 给了错误方案 | 背景不清楚 | 说「背景是：[说明]，在这个情况下怎么做」 |
| 结果和期望不一样 | 没说清楚"期望" | 用五要素模板重写需求 |
